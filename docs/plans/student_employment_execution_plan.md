# 大学生就业 Agent 产品执行总计划

## 1. 目标重述

### 北极星目标
帮助大学生把“不会求职、不会表达、不会筛岗位、不会跟进”变成一套可执行、可追踪、可持续优化的求职系统。

### 当前最正确的切入点
不是一开始做“大而全的就业平台”，而是先用 `AI 简历定制 + 求职工作台 + JD 匹配反馈` 打穿第一个高频刚需场景。

### 当前项目的实际定位
基于现有代码，项目已经具备以下雏形：

- 简历生成与优化
- JD 匹配评分
- 个人候选人工作台
- 岗位机会导入与匹配打分
- 申请记录与面试准备入口

结论：
项目的正确方向不是“通用 ReAct Agent 框架”，而是“面向大学生就业的求职 Agent 产品”，简历是第一入口，不是最终边界。

---

## 2. 产品总战略

### 产品一句话
做一个真正帮大学生拿到 offer 的求职 Agent，而不是一个只会润色文案的简历工具。

### 用户分层

#### 第一优先用户
- 统招本科/硕士应届生
- 缺少高质量求职辅导
- 目标是互联网、软件、数据、产品、运营、教培、商科管培等白领岗位
- 有基础经历，但不会包装，不会针对 JD 定制，不会持续跟进

#### 第二优先用户
- 0-3 年工作经验的校招生/初职场人
- 需要转岗、跳槽、跨城市、跨市场求职

#### 暂不优先
- 蓝领招聘
- 资深猎头场景
- 企业 ATS SaaS
- 复杂多租户 B 端平台

### 价值主张

对学生：
- 不再只给建议，而是直接帮他完成求职动作
- 从“写简历”延伸到“找岗位、定制、投递、复盘、面试准备”

对传播：
- 输出结果足够直观，适合在小红书、B站、公众号、社群传播
- 简历前后对比、岗位匹配分、面试准备卡片都天然适合分享

对留存：
- 求职不是一次性行为，是连续 4-12 周的过程
- 工作台和任务流天然比单次生成工具更容易做留存

## 3. 业务线梳理

## 3.1 核心业务主线

### 业务线 A：简历诊断与定制
这是当前最成熟、最该继续深挖的主线。

核心功能：
- 简历基础信息录入
- 原始简历解析
- JD 匹配诊断
- 项目/经历重排
- Bullet 改写
- 模板与排版生成
- 一页版/两页版控制
- 导出简历

用户结果：
- 产出一份能投递的简历

这是最适合做冷启动和传播的爆点入口。

### 业务线 B：岗位匹配与投递决策
简历之后，用户的第二个刚需不是“继续改简历”，而是“我应该投什么、先投什么、这份简历适不适合这个岗位”。

核心功能：
- 导入 JD
- 抽取岗位关键信号
- 候选人与岗位匹配评分
- 缺失信号提示
- 推荐对应简历版本
- 生成下一步动作建议

用户结果：
- 明确哪些岗位值得投，避免海投

### 业务线 C：求职工作台
这是把工具变成产品的关键。

核心功能：
- 候选人画像
- 目标岗位管理
- 简历版本管理
- 机会池管理
- 投递记录
- 面试准备列表
- 关键记忆和偏好存储

用户结果：
- 用户不再是一次性来“生成文案”，而是在这里持续推进求职

### 业务线 D：面试准备与复盘
这是简历之后最自然的第二增长曲线。

核心功能：
- 根据简历和 JD 自动生成自我介绍
- 生成项目深挖问答
- 生成八股/行为面试问答
- 面试复盘记录
- 根据面试反馈反推简历和岗位策略

用户结果：
- 提高从投递到面试、从面试到 offer 的转化率

## 3.2 推荐的产品边界

### 现在就做
- 简历定制
- JD 匹配
- 岗位评分
- 简历版本管理
- 投递跟踪
- 面试准备基础版

### 第二阶段做
- 职位抓取/职位导入增强
- 多岗位批量比较
- 面试复盘闭环
- 求职周报
- 导师/辅导服务结合

### 暂时不要做
- 通用聊天机器人
- 多 Agent 炫技编排
- 企业招聘后台
- 复杂社交社区
- 非求职主链路的大量外围功能

## 4. 如何做成爆款项目

### 爆款核心不是“功能多”
而是三件事：

1. 结果足够强
2. 分享足够自然
3. 留存足够高

### 爆款路径

#### 第一步：拿下最可传播的单点结果
先做出 3 个高传播结果页：

- 简历诊断报告
- JD 匹配报告
- 简历前后对比结果

要求：
- 一眼能看出改进
- 非专业用户能理解
- 截图就能传播

#### 第二步：把“生成”升级成“行动”
用户不满足于“你给我写得不错”，而是想知道：

- 我接下来投谁
- 先投哪几个
- 哪份简历最适合
- 我为什么没过

所以产品要从“内容生成器”升级到“求职行动助手”。

#### 第三步：围绕 offer 转化做口碑
真正能让产品形成裂变的不是“文案好”，而是：

- 用户拿到更多面试
- 用户面试反馈明显变好
- 用户拿到 offer

所以每一条业务线都要回到一个核心指标：
是否提高“投递 -> 面试 -> offer”的转化率。

### 推荐传播钩子

- 30 秒生成面向某 JD 的定制简历
- 一键看出简历为什么过不了筛选
- 自动把散乱经历包装成可投岗位的简历
- 自动判断这份岗位值不值得投
- 自动生成面试自我介绍和项目回答

### 推荐最小商业化路径

- 免费：基础诊断、有限次数 JD 匹配、工作台基础功能
- 付费：深度定制、导出高质量版本、多版本管理、面试准备套件
- 后续：训练营/1v1辅导/企业合作

## 5. 产品路线图

## Phase 0：收敛方向

### 目标
把当前项目从“过渡期混乱状态”收敛成一个明确产品。

### 输出

### 本阶段决策

- 第一入口：AI 简历定制
- 第一主链路：画像 -> JD -> 简历定制 -> 岗位评分 -> 投递跟踪
- 老的通用 Agent 方向降级为底层能力，不再作为对外叙事中心

---

## Phase 1：打磨爆点 MVP

### 目标
做出一个用户愿意分享的“简历 + JD 定制”爆点产品。

### 核心功能

#### F1. 候选人基础画像
- 姓名、专业、毕业时间、目标岗位、目标城市
- 项目/实习/竞赛/校园经历录入
- 目标市场偏好

#### F2. JD 导入与解析
- 粘贴 JD 文本
- 提取岗位关键词
- 抽取职责、要求、加分项
- 输出岗位画像

#### F3. 简历诊断报告
- ATS 风险
- 关键词覆盖率
- 量化成果密度
- 可读性
- 岗位匹配度

#### F4. 简历定制生成
- summary 改写
- 项目/经历排序
- bullet 改写
- 技能表重组
- 一页版控制
- 导出文档

#### F5. 对比结果展示
- 原版 vs 优化版
- 改动说明
- 为什么这样改

### 成功标准
- 用户能在 10 分钟内产出第一份可投递简历
- 结果页有明显“前后差异感”
- 至少 3 个结果卡片适合截图传播

---

## Phase 2：从工具变成工作台

### 目标
让用户持续回来，而不是一次性生成后离开。

### 核心功能

#### F6. 简历版本管理
- 每个目标岗位生成一个简历版本
- 标记状态：草稿/已投递/已使用/待优化
- 记录适用岗位和市场

#### F7. 岗位机会池
- 导入岗位
- 打分排序
- 推荐最佳简历版本
- 标记是否值得投递

#### F8. 投递跟踪
- 公司、岗位、时间、进度
- 下一步动作
- 是否需要跟进

#### F9. 求职记忆
- 用户偏好的岗位和城市
- 最强项目
- 面试常被问到的问题
- 不同岗位的投递反馈

### 成功标准
- 用户一周内至少回来 2 次
- 至少管理 5 个岗位机会
- 至少使用 2 个简历版本

---

## Phase 3：提高面试转化

### 目标
把产品价值从“获得投递材料”推进到“提高面试率和 offer 率”。

### 核心功能

#### F10. 自我介绍生成
- 30 秒版本
- 1 分钟版本
- 适配不同岗位版本

#### F11. 项目深挖问答
- 按项目生成高频问题
- 给出 STAR 结构回答
- 针对 JD 做追问

#### F12. 行为面试准备
- 围绕领导力、协作、压力、冲突、成长
- 自动关联候选人真实经历

#### F13. 面试复盘
- 记录问到的问题
- 标记没答好的点
- 反推需要补强的简历信号

### 成功标准
- 用户能生成一套可练习的面试材料
- 简历和面试内容开始联动
- 用户能形成“投递-面试-复盘-再优化”的闭环

## 6. 功能优先级与验收标准

## P0：必须马上做好的功能

### 1. 简历输入模型统一
问题：
当前简历数据结构存在历史包袱，字段命名不够统一。

要求：
- `resume_data` 结构统一
- `experience/experiences` 等兼容逻辑收敛
- 建立稳定 schema

验收：
- 同一份数据从输入到评分到导出不再需要隐式兜底

### 2. JD 解析与评分稳定
问题：
现在有启发式评分，但还不够像“产品级岗位分析器”。

要求：
- 明确关键词提取逻辑
- 输出职责、要求、加分项
- 区分硬性要求和软性信号

验收：
- 同一个 JD 连续分析结果基本一致
- 用户能理解评分原因

### 3. 简历定制结果可解释
问题：
用户不只需要结果，还需要知道为什么这样改。

要求：
- 输出修改说明
- 标注新增关键词
- 标注删除或弱化内容

验收：
- 用户能对优化结果建立信任

### 4. 导出质量稳定
问题：
导出是最终交付物，任何不稳定都会直接伤害口碑。

要求：
- 模板一致
- 排版可控
- 一页版不炸版

验收：
- 10 份样例简历导出结果可读、结构稳定

## 7. 技术与产品的收敛原则

### 原则 1
产品叙事只讲“大学生就业 Agent”，不再对外主讲“通用 Agent 框架”。

### 原则 2
一切功能都围绕求职主链路，不做炫技型功能。

### 原则 3
能用规则解决的先用规则，Agent 只解决高变化、高表达需求。

### 原则 4
先把单用户单工作台跑顺，再考虑平台化和高并发。

### 原则 5
导出物、结果页、评分解释优先于底层架构炫技。

## 11. 执行纪律

### 接下来所有开发都遵循以下顺序
1. 先判断是否直接服务求职主链路
2. 再判断是否提高转化、留存或传播
3. 再决定是否值得开发

### 明确不做
- 与求职主链路无关的泛 Agent 能力扩展
- 过早平台化
- 过早多端化
- 过早复杂多 Agent 编排

### 明确先做
- 结果质量
- 业务闭环
- 结构统一
- 工作台化

## 12. 接下来我们的执行方式

从现在开始，默认按以下优先级推进：

1. 收敛数据结构
2. 打磨简历诊断与定制结果
3. 建立岗位评分和简历版本管理
4. 完成求职工作台闭环
5. 补上面试准备链路

### 第一批具体开发任务
- 统一 `resume_data` schema
- 设计 `jd_analysis` 统一结构
